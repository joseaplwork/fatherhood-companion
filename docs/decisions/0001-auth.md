# ADR 0001 — Authentication and User Identity

## Status

Decided

## Context

The platform requires user authentication, session management, and a stable user identity key
that all application data can reference. Building these from scratch is out of scope.

Children profiles are lightweight and tightly coupled to the parent account. They do not need
their own auth and do not need to be queried independently across users.

## Decision

Use **Clerk** for authentication and user identity.

- Clerk manages: sign-up, sign-in, sessions, email, full name, and avatar.
- All application database records reference the Clerk user ID as the owner key.
  There is no separate `users` table in the application database.
- **Child profiles are stored in Clerk private metadata**, keyed by a system-generated
  numeric ID. Child data is read and written exclusively through the Clerk API and is
  never stored in the application database.
- The `UserProfile` entity in the application database extends Clerk identity with
  app-specific fields (onboarding state, subscription tier, bio, etc.).

## Consequences

- No custom auth infrastructure to maintain.
- Clerk's private metadata is the single source of truth for children. Any feature that
  needs child data reads it from the Clerk session or the Clerk backend API.
- Child numeric IDs (from Clerk metadata) are used as reference keys in `CalendarEntry`
  and any other entity that links to a specific child.
- If child data ever needs to be queried server-side without a user session context
  (e.g., admin tooling), it must go through the Clerk backend SDK, not the application DB.
